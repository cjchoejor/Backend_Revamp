## 5. Self-Check Process — What Happened and What Was Learned

### 5.1 First Self-Check (Memory-Based — Insufficient)

After writing, Claude ran an initial self-check from memory rather than against live source files. This produced 6 findings but was structurally insufficient — it checked against Claude's recollection of what it had written and what the sources said, not against the actual files.

**This is the same failure mode documented in MOM-ARCH-2026-008** (Gate 1 regression where sections were written from memory rather than active source documents). The Architect identified this before corrections were applied.

### 5.2 Source-Grounded Self-Check (Correct Process)

At Architect direction, Claude ran a proper source-grounded check. Process:

1. Read the full written Part 5 file in chunks using the `view` tool (not from memory)
2. Read §72 matrix exactly, row by row, checking every stage cell against the corresponding policy in the written file
3. Read §70A M.1–M.14 against Policies 71–72 clause by clause
4. Read §70B M.1–M.14 against Policies 73–75 clause by clause
5. Read §70C M.1–M.14 against Policies 76–77 clause by clause
6. Read Part 4 engine method signature table against every engine delegation in the written file
7. Spot-checked enum values against Part 2 §2.1.3

This produced 17 findings — 11 more than the memory-based check.

**Key lesson recorded:** The correct self-check process is always source-grounded. The view tool must be used to read both the written file and the source simultaneously. Checking from memory is not self-checking — it is rechecking what was already in context when the error was made.

## 7. Deliberations and Decisions

### Deliberation 1 — RF-P5-01 + RF-P5-06 (Coupled)

**Question:** Policy 2 (Availability group) and Policy 47 (DEFICIENT condition group) both had S1 scope. §72 has two rows pointing at the same enforcement event (AvailabilityService DEFICIENT annotation). Two options presented:
- Option A: Give Policy 47 its own full definition, sharing the enforcement point
- Option B: Policy 47 becomes a formal cross-group reference note; one definition, one enforcement point

**Decision:** Option B confirmed by Architect.

**Rationale:** One enforcement point, one policy definition. Two definitions for the same `AvailabilityService` annotation step would mislead implementation — an LLM could implement two annotation paths where only one exists.

**Changes applied:**
- Policy 2 narrowed to S1 only; S5 language removed
- Policy 47 replaced with formal cross-group reference note stating: Policy 2 is authoritative; no separate enforcement point exists; implementation must not create one; S5 scope belongs exclusively to Policy 48
- Summary table Policy 2 corrected to S1 only

---

### Deliberation 2 — RF-P5-02

**Question:** Policy 56 used "Deferred" as a fourth decision type violating §5.1.1's tri-decision model. Should the tri-decision model be relaxed for Policy 56, or should the deferral path be expressed within the model?

**Context provided:** The no-show determination has three operationally distinct paths — (1) no-show confirmed immediately, (2) written confirmation pending with governed timer, (3) process too early to initiate. Path 2 triggers completely different downstream behaviour from Path 1.

**Decision:** Tri-decision model kept intact. Policy 56 APPROVED decision expressed as two explicit sub-paths:
- Sub-path 1 (No-show determined): `NoShowDeterminationRecord` created; Policy 57 financial mechanics fire immediately
- Sub-path 2 (Written confirmation pending): entry enters `AWAITING_WRITTEN_CONFIRMATION`; `TimerEngine.register()` called; two outcomes — written confirmation arrives (no-show abandoned, no record created) or timer fires (Sub-path 1 mechanics execute at that point)

**Rationale:** The deferral path is a consequence of FOM's determination authority, not a separate policy decision type. `AWAITING_WRITTEN_CONFIRMATION` is a state machine path already defined in Part 3 §3.16 — it should not be re-expressed as a policy decision category.
