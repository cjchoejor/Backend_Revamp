## Schema File Convention

All model definitions in this part constitute the content of a single `schema.prisma` file. Prisma schema DSL is used throughout. No raw SQL. No pseudocode. Every model definition in this part is valid Prisma syntax intended to be composed into one schema file.

The canonical enum registry is defined once in §2.1 and applies throughout. Model definitions in subsequent sections reference these enums directly — they are not redefined per section.

## §2.1 — Schema Design Principles

### 2.1.1 Governing Doctrine

The following rules apply to every model definition in this part without exception. A model definition that violates any of these rules is architecturally wrong regardless of whether it compiles.

**Append-only doctrine.** Operational records are never destructively edited. Corrections to committed records are additive layers — a new record, a new event, a new line. No `UPDATE` that changes the meaning of a committed fact is permitted. The original record persists.

**Immutability by layer.** Records become progressively immutable as they pass through stages. The commitment snapshot at S4 (the Reservation record) is the first major immutability boundary. Records that are immutable from creation carry no `updatedAt` field and no editable fields after the initial `CREATE`. Schema-level constraints enforce this where possible.

**NOT NULL on all audit fields.** `actor_id`, `entity_id`, `operation`, and `timestamp` on every trace event row must be `NOT NULL` at the database level. A trace event row with a null actor is a structural defect — it is not a recoverable data problem.

**Layered control hierarchy reflected in schema.** Canon truths are enforced at schema constraint level (NOT NULL, foreign key, unique constraints). Policy rules are enforced at service level. Configuration values are never hardcoded in schema — they are read from configuration tables at runtime.

**Prisma as source of truth.** The `schema.prisma` file is the authoritative schema definition. Migrations are generated by `prisma migrate`. No schema changes are made by editing migration files directly. No raw SQL DDL is used to create or alter tables outside the Prisma migration toolchain.

**Single-tenant.** No `tenant_id` column exists on any table. No multi-tenancy partitioning. This system is deployed for one property.

**§71 mutation rules obligation.** Every record type's create/update/amend/seal/archive rules are derived from §71 (Canon Block 11). These rules are stated as annotations on each model in the sections below. Where a mutation rule can be enforced at schema level (e.g., no `updatedAt` field on an immutable record), it is. Where a mutation rule requires service-level enforcement (e.g., "only additive layers permitted after S4"), it is stated as a constraint annotation for the service gate writer.

**§74 stage-to-record matrix derivation obligation.** Every record type's lifecycle columns support the create/read/update/seal operations specified in §74 (Canon Block 11 — Stage-to-Record Matrix). The matrix is reproduced in reduced form in §2.1.3 below as a derivation reference. Service and state machine writers derive their per-stage operations from the full §74 matrix.

## §2.2 — Entity Hierarchy: Inquiry / Entry / Segment

### 2.2.1 Overview

The entity hierarchy is the foundational three-level structure of the system. Every operational record anchors to one of these three levels. The hierarchy is:

- **Inquiry** — the commercial container. Groups one or more entries under a single guest identity and engagement context.
- **Entry** — the lifecycle unit. Progresses independently through S1–S9. One Inquiry may contain multiple entries (combined engagements), each with its own use type and independent lifecycle.
- **Segment** — the re-entry layer. One segment per pass through the stages. A new segment is created on re-entry; prior segments are sealed and read-only.

**Mutation rules (from §71):**
- Inquiry: created at S1; custodian and contact fields editable while active; sealed when all child entries are terminal; archived on closure.
- Entry: created at S1; provisional fields editable until commitment boundary at S4; new segment on re-entry; artifacts seal per stage exit rules; archived on S9 closure.
- Segment: created on re-entry; only current segment's current stage is editable; sealed when superseded or entry closes.
- Availability Configuration: created at S1; editable during S1 including DEFICIENT acknowledgements; sealed on S1 exit; remains recallable.

## §2.3 — Inventory: Claim State and Physical State

### 2.3.1 Two-Model Architecture

Inventory is governed by two independent models evaluated separately (§28):

- **Model 1 — Inventory Claim State:** Evaluated against a date range, not point-in-time. Tracks the commercial reservation status of each room or space. States defined in §2.1.3 enum registry.
- **Model 2 — Physical State:** Tracks the physical condition of a room or space. The DEFICIENT flag is an overlay on physical state — not a separate state.

These two models are never merged. A room may be physically clean but commercially COMMITTED_HELD. A room may be physically DEFICIENT but commercially FREE. Both dimensions are evaluated independently before assignment.

## §2.4 — Financial: Folio / Invoice / Payment

### 2.4.1 Financial Record Architecture

The financial layer is built on sequential immutability. The governing rules:

- The Provisional Folio (S3) accepts only proforma invoices and advance payment records. It does not accept live charges.
- At S6, the Provisional Folio converts to a Live Folio. This conversion is an explicit audited event, not an update.
- No folio line is ever edited after posting. Corrections are additive: credit notes, Level 2 additive records (FOM), Level 3 correction entries (GM).
- Post-closure backdating is prohibited: post-closure records carry their actual transaction date, not the original stay date (§29.5).
- BTN is the authoritative currency. Foreign currency payments carry `foreign_currency_amount`, `btn_equivalent`, and `exchange_rate`.
- Tax model is per rate plan: `ADDITIVE` (tax added on top) or `INCLUSIVE` (tax included). Rates are configurable — never hardcoded.
- Write-off requires GM authority and a recorded reason. Write-off is not automatic.

**Mutation rules (from §71):**
- Folio (Provisional): restricted edits — PIs and payment records only; converts to Live or closes as NO_SHOW_CLOSED; not amended.
- Folio (Live): charges posted until settlement; adjustment lines and credit notes added (never edit original lines); sealed at S9 closure; post-closure Level 2/3 adds layers.
- Invoice: not editable after dispatch; credit note or adjustment permitted — original never edited; sealed on dispatch.
- Payment Record: immutable from creation; not amendable — refund is a new payment-out event.
- Commission-Due Record: immutable from creation; not amendable.
- Credit Extension Ceiling Record: immutable from creation; ceiling revision adds a new record — original preserved.
- Credit Ceiling Threshold Event: immutable from creation.
- No-Show Determination Record: immutable from creation; created by FOM at S5 on no-show determination; anchors the no-show penalty FolioLine; not amendable.

## §2.5 — Rate and Pricing

### 2.5.1 Rate Plan Architecture

Rate plan selection follows a hardcoded priority order that the PricingPipelineEngine enforces. This order is not configurable:

1. Individual-level agreements
2. Promotional
3. Tier-level
4. Channel-level
5. Rack

This priority is PricingPipelineEngine behaviour. No configuration change may alter this order. The Minimum Sell Rate (MSR) is validated at both configuration time (Admin Console) and booking time (PricingPipelineEngine).

Deterrent rate plans are assigned automatically to Caution and Restricted tier clients. The guest is never informed that a deterrent rate applies (§30.3).

## §2.6 — Profiling: Guest / Agent / Corporate

The three profile types are persistent identity records. They are never closed — they evolve over time. Tier changes are layered events with recorded reason, not direct field edits.

## §2.7 — Communication

### 2.7.1 Communication Record Architecture

Communication records are immutable from creation. A sent message cannot be unsent — the record persists. Corrections produce new communication records. Subject line conventions (QUOTATION, CONFIRMATION, AMENDED, CANCELLED prefix) are enforced at the service layer, not the schema layer.

VOICE_NOTE message type triggers the human-only review routing path (§70C). AI systems are blocked at the routing layer from processing voice note content — this is a routing constraint, not a runtime check.

AI Draft Records: the AI agent produces a draft; a human must issue an explicit approval event before any outbound communication is sent. The AI agent may not approve its own drafts (§70B M.5 — absolute, no trust level overrides this).

**Mutation rules (from §71):** Communication records, AI Draft Records, Human Decision Records, Correction Records, AI Audit Supplement Records — all immutable from creation.

## §2.8 — Work Order

### 2.8.1 Work Order Architecture

The work order is a living control document. Every amendment is a new layered event — not an edit of the prior record. The work order is initiated at S1–S3 and lives through S8. Closure at S8 requires all to-do items to be either completed or cancelled with recorded reason.

Coordinator amendments are mediated by hotel staff and recorded — coordinators do not self-serve (§65 M.5). Consumption is tracked against allocations. Over-allocation requires an explicit acknowledgement event (§65 M.6).

**Mutation rules (from §71):** Work Order: amendments layered throughout lifecycle; every change is a new layer; sealed on S8 closure. Work Order To-Do Item: created at any stage S1–S7; status transitions (PENDING → IN_PROGRESS → COMPLETED/CANCELLED); completion evidence recorded.

## §2.9 — Dispute and Service Recovery

### 2.9.1 Dispute Architecture

A dispute is anchored to an entry. The `reservationId` is nullable — disputes may arise before S4. A dispute in OPEN or IN_PROGRESS state must reach terminal state before S9 closure proceeds (§53 M.12). This is a hard gate, not a warning.

`DISPUTE_EXHAUSTED` is explicitly not a valid state. A guest who declined all offered resolutions reaches CLOSED with reason "guest declined all offered resolutions" — not a DISPUTE_EXHAUSTED state (§53 M.12).

The `DisputeGateOverrideRecord` is independently queryable and is produced by the service layer — not the engine — when a GM override is applied at S7→S8. Override is not available at S8→S9 if dispute is in OPEN or IN_PROGRESS state.

Resolution Bundles (v2.4): one or more compensation actions approved as a single decision event. A new bundle is required if terms change — the original is preserved.

**Mutation rules (from §71):** Dispute: status transitions through lifecycle; resolution attempts layered; sealed on RESOLVED or CLOSED. Service Recovery Record: cannot be suppressed once created (§53 M.5); permanent record. DisputeGateOverrideRecord: immutable from creation. Resolution Bundle: items updated via additive execution events only — not direct field edits.

## §2.10 — Handoff

### 2.10.1 Handoff Architecture

Handoffs are formal transfers of operational obligation between roles or departments. Visibility to the receiving party does not constitute acceptance (§36.4) — acceptance is an explicit governed state transition. Completion requires defined fulfilment evidence per handoff type.

H1 auto-fulfilment: when reservations and front desk are the same team, the handoff is auto-fulfilled with an audit record. The infrastructure exists regardless.

H5 auto-fulfilment: when settlement is complete and no residual obligations exist.

REJECTED state: receiving party cannot accept due to blocking conditions. Produces a new routing event — not a terminal state.

**Mutation rules (from §71):** Handoff: status transitions (CREATED → ACCEPTED → FULFILLED); not amended — rejected handoffs produce new routing; sealed on fulfilment and closure.

## §2.11 — Quotation / Reservation / Confirmation

### 2.11.1 Commitment Architecture

The Reservation (commitment snapshot) is the most important immutability boundary in the system. Once created at S4:

- The commitment snapshot is immutable — no in-place edit.
- All commercial terms are frozen: rate, inclusions, conditions, credit ceiling.
- Any change after S4 requires a new segment with a new commitment snapshot.
- In-place editing of the commitment snapshot after S4 exit is the most dangerous architectural violation in the system (§45.18).

Quotations are versioned. A new version (Q-002) supersedes the prior version (Q-001). All versions are preserved. Sent quotations are not edited in place — a new version is created.

**Mutation rules (from §71):**
- Quotation: draft editable until sent; new version supersedes prior; sealed on send; all versions sealed on S2 exit.
- Hold (Speculative/Committed): timer and notes editable while active; not amended — released or upgraded; sealed on transition to CONFIRMED or on release.
- Reservation: not directly editable; amendment via re-entry; sealed on creation.

## §2.12 — Night Audit

### 2.12.1 Night Audit Architecture

One Night Audit Record per operating date. Immutable from creation. The night audit completeness check identifies missing expected charges as anomaly records — these are flagged for FOM review and are not auto-posted (§62 M.8).

## §2.13 — Timer and Scheduling

### 2.13.1 Timer Architecture

All governed timers are managed by the Timer Engine, which is implemented on pg-boss (PostgreSQL-native job scheduling). All expiry events are governed events with permanent audit records — silent expiry is a forbidden pattern.

The Processing Lock Record governs inventory concurrency during the booking workflow. It is transient — it informs but does not block other actors from viewing inventory. Hard expiry at TTL: no heartbeat or renewal mechanism. Reconfirmation creates a new ProcessingLockRecord with a new `lock_id` — it does not extend the existing record.

**Mutation rules (from §71):**
- Timer registry: created on activation; status transitions; fired or cancelled.
- Processing Lock Record: ACTIVE → EXPIRED (unconditional at TTL) or ACTIVE → RELEASED (booking completed). Expiry event permanently logged.

## §2.14 — Authentication / Session / Attribution

### 2.14.1 Session Architecture

Authentication is PIN-based fast user switching with JWT session tokens. Every session boundary event produces an immutable Session Event Record. Attribution is immutable: the `actor_id` on every record is the authenticated user at time of action. No "on behalf of" mechanism exists. No retrospective attribution change is possible.

Shared credentials are forbidden at the schema level. There are no group accounts, no shared credential accounts — `user_id` is always individual. This is a schema-level constraint enforced by the absence of any credential-sharing mechanism, not just a policy.

Session isolation per terminal: Terminal 1 session state does not bleed into Terminal 2.

The Session Event Record (v2.5, REV-B11-V25-01) is a formal canonical record type added at Canon v2.5. It is anchored to the staff user record. All four session boundary events — PIN_SWITCH, IDLE_AUTO_LOCK, MANUAL_LOCK, HARD_LOGOUT — produce a Session Event Record.

**Mutation rules (from §71, §71 v2.5):**
- Session Event Record: create-only; no update; no amendment; sealed at creation; archived with staff user record.

## §2.15 — Audit and Trace

### 2.15.1 Trace Event Architecture

The trace event table is the authoritative point-in-time reconstruction surface for the entire system. From trace events alone, every question in the Audit Reconstruction Map (§78) must be answerable. This is not a log table — it is a governed audit record.

NOT NULL constraints on `actor_id`, `entity_id`, `operation`, and `timestamp` are enforced at the database level. A trace event row that lacks any of these four fields is a structural defect.

The correlation identifier links events in a chain: inbound message → AI classification → draft creation → human decision → outbound send. This chain must be traceable without any inference.

**Mutation rules (from §71):** Communication Records — immutable. Trace events: append-only; no row is ever updated or deleted.

### 2.15.2 Prisma Models

```prisma
model TraceEvent {
  // Append-only. No UPDATE. No DELETE. Ever.
  id              String    @id @default(uuid())
  eventType       String    // canonical event type name from §77 event catalogue
  actorId         String    // NOT NULL — always a specific actor (L0–L4 or EXT)
  actorLevel      ActorLevel // NOT NULL
  entityType      String    // NOT NULL — canonical record type name from §1.6.19
  entityId        String    // NOT NULL
  operation       String    // NOT NULL — CREATE | UPDATE | SEAL | CLOSE | TRANSITION | etc.
  payload         Json      // structured payload — never freeform string
  timestamp       DateTime  // NOT NULL — @default(now()) is a minimum; caller sets explicitly
  stageContext    Stage?    // stage at time of event; null for session and config events
  segmentContext  String?   // Segment.id if applicable
  correlationId   String?   // links events in a chain
  inquiryId       String?   // top-level anchor for cross-record queries
  entryId         String?   // operational record anchor
  createdAt       DateTime  @default(now())

  // No relations defined — TraceEvent is a sink. It references entity IDs
  // as plain strings to avoid cascading referential constraints that could
  // prevent appending events for closed or archived records.

  @@index([entityType, entityId])
  @@index([correlationId])
  @@index([actorId])
  @@index([timestamp])
  @@index([inquiryId])
  @@map("trace_events")
}
```

## §2.19 — Incident and Emergency

### 2.19.1 Incident Architecture

Incident records anchor to an entry and a room. For DEATH and SECURITY incidents, the room transitions to BLOCKED and requires the `cleansing_ritual_completed` flag before return to service (§69 M.8). This flag is on the Room model (§2.3) — not on the incident record.

Lost and found records carry a configurable retention period. Guest association is nullable — found items may not be attributable to a specific guest.

**Mutation rules (from §71):** Incident Record: not explicitly stated as immutable — status transitions permitted (OPEN → CLOSED); may remain open for legal reasons. Lost and Found: status transitions (FOUND → RETURNED | DISPOSED).

## §2.20 — Database Migration Strategy

### 2.20.1 Migration Toolchain

The migration toolchain is `prisma migrate`. All schema changes are expressed as Prisma schema modifications. Migration files are generated by Prisma from schema diffs — they are not written by hand. No raw SQL DDL is used to create or alter tables outside the Prisma migration toolchain.

### 2.20.2 Existing Codebase Considerations

This system is a refactoring of a working codebase, not a greenfield deployment. The migration strategy must handle three categories of change:

**Category A — New tables.** Tables that do not exist in the current codebase. These are created by new Prisma models. `prisma migrate` generates `CREATE TABLE` statements. No existing functionality is affected.

**Category B — Column additions to existing tables.** New columns added to tables that already exist. All new columns that are `NOT NULL` must be introduced with a default value or a two-step migration: (1) add column as nullable, (2) backfill, (3) add NOT NULL constraint in a subsequent migration. Skipping this sequence will fail on tables with existing rows.

**Category C — Structural alignment of existing tables.** Tables that exist in the current codebase but whose structure diverges from the Prisma schema. Each divergence is a migration risk. The strategy is: (1) map current table structure against the Prisma model, (2) identify added fields, renamed fields, type changes, and removed fields, (3) handle each category conservatively — rename with explicit `@@map` before dropping the old column, never drop a column in the same migration as creating its replacement.

### 2.20.3 Backwards-Compatibility During Migration Period

During the migration period — defined as the time between the first schema migration and full service layer alignment — the following rules apply:

- No column is dropped until all code that references it has been removed or redirected.
- Foreign key constraints are added after data is confirmed consistent — not before.
- Unique constraints on existing data require a data audit before the migration runs.
- New NOT NULL columns on existing tables must carry database-level defaults for the migration period. These defaults are removed once all code paths populate the field explicitly.

### 2.20.4 Rollback Strategy

Every `prisma migrate` deployment has a corresponding rollback plan:

- For new tables: rollback is `DROP TABLE` — safe when the table is empty or the data is recoverable.
- For column additions: rollback is column drop — safe only if no data has been written to the new column in production.
- For structural alignment: rollback requires the reverse migration to be tested in a staging environment before the forward migration is applied to production.

Rollback plans are documented in the migration file's comment block. Every migration file that touches an existing table carries an explicit rollback instruction.

### 2.20.5 Seed Data Strategy

Configuration tables (§2.17) require seed data before the system is operational. The seed strategy is:

- Seed data is defined in `prisma/seed.js` (Node.js, `require()` syntax consistent with the existing codebase).
- Seed data targets configuration tables only — it does not create operational records.
- Seed data is idempotent: running the seed script on a database that already contains seed data produces no duplicates and no errors.
- Configurable thresholds, authority bands, and default values are defined in Appendix C (Seed Data Specification) and implemented in the seed script.
- The seed script runs as part of the deployment pipeline after `prisma migrate deploy`.
