Both sessions are part of the decision record. Excluding them from the declared source list would make the derivation chain inaccurate.

### RF-2 — "Inquiry" vs "Engagement" Canonical Term

**Nature:** Terminology lock — deliberation required.

**Flag:** MOM-ARCH-2026-007 §5.1 records a deliberated rename of "Inquiry" to "Engagement" as the canonical term for the commercial container entity. The Canon v2.4 (all 11 blocks) consistently uses "Inquiry" throughout — every stage charter, every cross-stage mechanism, every entity definition. Gate 1 draft used "Inquiry" and flagged the conflict for Architect resolution.

**Deliberation:** The Canon is the source of truth in this system. Its consistent use of "Inquiry" across all 11 blocks is authoritative. A MOM decision that was never absorbed into the Canon is an incomplete decision — per Canon §3, the Canon supersedes the MOM where the two conflict. The MOM-007 rename was deliberated but was not carried forward into the governing document. Using "Engagement" in the spec while the Canon uses "Inquiry" would create a misalignment between the governing document and every downstream implementation artifact.

If the rename is to be executed, the correct path is a Canon revision replacing "Inquiry" with "Engagement" throughout all 11 blocks. That is a separate formal decision requiring its own MOM and Canon revision. It is not a Gate 1 decision.

**Decision:** CONFIRMED by Architect. **"Inquiry" is the locked canonical implementation term.** The MOM-007 rename is not active. The Prisma model is `Inquiry`. The primary key is `inquiry_id`. All foreign keys, service class names, and API routes that reference the commercial container use "Inquiry".

**Change applied:** No text change required in the spec — draft already used "Inquiry" throughout. The RF-2 note in §1.6 preamble is retained as a naming convention confirmation rather than a flag.

### RF-3 — Session Event Record Canonical Status

**Nature:** Canon gap resolution — deliberation required.

**Flag:** §34.4 (Authentication, Attribution, and Session Control) requires that session boundary events — PIN switch, idle auto-lock, manual lock, hard logout — are recorded with timestamp and actor. This language is identical to the language the Canon uses for all other governed events that produce canonical records. However, "Session Event Record" did not appear in §70 (Canonical Record Types), which is the authoritative register. The Gate 1 draft listed it as a canonical record type in §1.6.19 and flagged the absence from §70 for resolution.

**Deliberation summary presented by Claude:**

The Canon's language in §34.4 is unambiguous — session events are recorded with timestamp, actor, outgoing user, incoming user (for PIN switch), and event type. That is not the language of an implementation log entry; it is the language of a governed audit record. The Canon applies the same "recorded event" doctrine to session events as it does to parking events, expiry events, and handoff state transitions — all of which produce canonical records in §70.

The absence from §70 reads as an omission, not a deliberate design choice. The practical consequence of leaving session events as implementation details is that the Part 2 gate writer would have no schema guidance, would encounter §34.4 with no derivation path, and would either invent a schema (violating the derivation obligation) or surface it as a Part 2 Category 1 blocker.

Three operational requirements depend on queryable session history: attribution integrity verification, shift boundary governance, and FOM session visibility (escalation frequency and session management reports). These require a first-class record with a defined schema, not a generic debug log.

**Decision:** CONFIRMED by Architect. **Session Event Record is a formal canonical record type.** The absence from §70 is an omission. It is resolved via Canon v2.5 addition rather than retroactively modifying v2.4.

**Changes applied:**

1. CANON-V2.5-CHANGESET.md produced with two entries:
   - REV-B11-V25-01: Session Event Record added to §70 — four event types (PIN_SWITCH, IDLE_AUTO_LOCK, MANUAL_LOCK, HARD_LOGOUT), defined fields, immutable from creation, anchored to staff user record
   - REV-B11-V25-02: Session Event Record mutation rules added to §71 — create-only, no update, no amendment, sealed at creation, archived with staff user record

2. DEV-SPEC-001-Part1.md §1.6.19 updated — Session Event Record added to canonical record type names list with v2.5 marker.

3. Part 2 schema gate obligation created: Session Event Record requires a Prisma model at the Part 2 gate. Declared Canon source for that model: §34.4 and Canon v2.5 REV-B11-V25-01.
