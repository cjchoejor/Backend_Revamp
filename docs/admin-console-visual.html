Admin Console · Implementation Reference

The Admin Console configures.
It does not operate.

A single off-axis governance surface that owns every parameter the nine operational stages read at runtime — and writes none of the records they produce. This is the visual reference for what gets configured, by whom, where each value flows, and how it stays correct.

Frame 01 · Surface Boundary

One direction of motion. Always.

The Admin Console writes to configuration tables. The nine operational stages read from those same tables at runtime. No admin surface ever creates, updates, or closes an operational record. Cross-direction writes are structural defects, not business decisions.

What Admin Owns

Configuration tables only. Hotel profile, departments, staff registry, roles, room types, rate plans, seasons, packages, thresholds, modes, policies, templates, schedules, OTA & AI configuration.

What Admin Never Touches

Every operational record. Inquiry, Quotation, Hold, Reservation, Folio, Invoice, WorkOrder, HandoffRecord, NightAuditRecord, NoShowDeterminationRecord — admin code cannot create, update, or close any of them.

The One Exception

AgentProfile.commissionRate is the lone admin write into an operational record — narrowly scoped to PostStayAndGovernanceService. Everything else stays inside the boundary.

Frame 02 · Authority Model

Every admin write is locked at L4.

The L4 designation is not a UI access list. It is woven through every admin service, middleware, and route. L1 Front Desk, L2 FOM, and L3 GM-in-operations cannot reach any admin write path — the rejection returns AuthorizationError 403, not a generic 401, so the caller's authority level is recorded.

Authority Rules

  • Custom mode activationL4 GM specifically — not any L4. A custom mode cannot be activated by anyone else, even an actor otherwise authorised to save mode configurations.
  • Audit visibility readsA subset of read paths extends to actors below L4 via role-permission mapping — own session events, scoped audit logs, readiness summary.
  • Individual attributionEvery admin action carries the actorId of the specific administrator. There is no shared admin account.
  • Per-role session configIdle-lock timeout, hard-logout timeout, and manual-lock availability are configured per role — every defined role has its own RoleSessionConfig. No shared timeouts.
  • Single-tenant namespaceOne deployment, one configuration namespace. No tenantId column anywhere in admin schema.

Frame 05 · Configuration Lifecycle

Time, not state, is the lifecycle.

Configuration has no stage gate. A value is active while now() falls inside its [effectiveFrom, effectiveTo) window. Saves don't edit — they supersede. The previous row's effectiveTo is set to the new row's effectiveFrom, both writes happen in one transaction with the audit event, and history is retained forever.

Three States

ACTIVE (effectiveTo = NULL), SUPERSEDED (effectiveTo set), HISTORY (retained forever). At most one ACTIVE row per configKey.

Atomic Supersession

Update prior effectiveTo, insert new row, write TraceEvent — all three writes in one Prisma transaction. If any rolls back, all roll back.

Optimistic Locking on Registries

Registry rows carry a version: Int field. The update request must match the current version, or the second writer is rejected with ConcurrentEditingError. No silent overwrites.

Frame 06 · Mode Lifecycle

Modes have a validation gate the rest of configuration doesn't.

A mode defines the operational route — which stages run, what auto-fulfils, what features depend on which services. Because a bad mode could silently bypass policy or dead-end the state machine, every save runs through the ModeValidationEngine against three constraint categories before it's allowed to persist. The current state lives on the row itself as a typed state field on ModeConfiguration.

Eight Seed Modes (ship Active)

  1. New Booking
  2. Room Change
  3. Rate Revision
  4. Date Extension
  5. Early Departure / Cancellation
  6. Billing Model Change
  7. Guest Composition Change
  8. Complaint Resolution / Goodwill
Predefined modes ship ACTIVE on initialisation. They may be customised through supersession, but they may not be deleted. Custom modes built by an administrator follow the full DRAFT → VALIDATED → ACTIVE path and require the L4 GM specifically to activate.
Frame 07 · Readiness Gate

A stage is not live until its configuration is complete.

At system bootstrap, the ReadinessGateEngine evaluates every configuration surface required by every readiness group. It collects all failures, never fail-fast — an administrator must see every gap in one configuration session, not one restart at a time. There are no fallback defaults. Absence of a required surface raises MissingConfigurationError; the operation halts.

Validity Criteria (per surface)

Surface record exists · all mandatory fields populated · all referential dependencies resolvable. Per-surface logic delegated to the owning service's validateForReadiness() method.

No Fallback Defaults

If a required surface is missing, no service may substitute a hardcoded default. Workers raising MissingConfigurationError halt the operation; they don't proceed with an assumed value.

When It Runs

At system bootstrap before operational requests are accepted, and on demand via the admin ReadinessService when an administrator requests re-validation.

What It Returns

For each readiness group: LIVE or BLOCKED, plus the full failure list. Overall status: ALL_LIVE or SOME_BLOCKED.

Frame 08 · Structural Policies

Seven structural policies enforce the surface — at seven different layers.

These are not preferences. Each policy carries a specific applicability, a specific enforcement mechanism, and a specific error contract. The layer at which each is enforced is part of the policy itself.

A1
Strict Separation from Operational Code

Admin code never imports operational services. Operational code never imports admin services. The two layers live in separate top-level directories with a build-time boundary check. The forbidden-patterns list explicitly carves out agent-profiles — the one narrow write into an operational table — so it isn't flagged as a violation.

Enforced at: Build-time / review-time
A2
Audit Event on Every Configuration Save

Every CREATE, UPDATE, and DELETE writes a TraceEvent in the same transaction as the change. If either write fails, both roll back. There is no path to "save without audit."

Enforced at: Service layer · transaction
A3
Default Values Visually Distinguished

Every UI surface that renders a configurable value shows a non-intrusive marker when that value is a seeded default and has not been administrator-reviewed. Indicator disappears on first conscious save.

Enforced at: UI layer
A4
Mode Configuration Validated at Save Time

Every mode save runs through the ModeValidationEngine for policy, state-machine, and service-dependency constraints. Save proceeds only on ACCEPTED. L4 cannot override.

Enforced at: Service layer · via engine
A5
Configuration Cannot Disable a Required Control

Saves that would disable exit-evidence checks, timer governance, AI approval gates, audit writes, or financial immutability are rejected with ConfigurationViolationError. Tunes parameters; does not remove controls.

Enforced at: Service layer · check helper
A6
Single-Tenant Namespace

No tenantId column on any admin table. No partition filter on any admin query. No property selector on any admin UI. Multi-property is a future scope extension, not a runtime feature flag.

Enforced at: Schema review · code review
A7
L4 Authority on Every Admin Write

Every write route registers requireL4() after auth, before validation. Non-L4 callers get AuthorizationError 403 with their authenticated level recorded. A read-only middleware requireAdminRead() handles audit-visibility extensions.

Enforced at: Middleware layer
∑
All seven enforced together. None redundant.

A1 governs code-layer separation; A7 governs runtime authorisation. Both are required — the build-time check protects against misimports, the middleware check protects against runtime bypass. Every policy has its own enforcement layer because each catches a different class of error.

Net effect: Defence in depth across 7 layers
Frame 09 · Engines

Two pure engines. No state. No side effects.

The Admin Console introduces only two engines. Both take typed inputs, return typed outputs, and own no state of their own. They orchestrate; the owning admin services hold the per-surface truth.

ReadinessGateEngine

readiness · evaluation · all-failures-collected
Purpose
Decide whether every required configuration surface for every readiness group is populated, valid, and fit to be read by operational code.
Invocation
System bootstrap (synchronously, before traffic) and on-demand via ReadinessService.runReadinessCheck.
Input
{ groups[], asOf } — supports historical audit (asOf in the past).
Output
{ groupResults[], overallStatus } with every MissingConfigurationError collected.
Hardcoded
Collection-not-fail-fast · no fallback defaults · readiness manifest is code, not config.

ModeValidationEngine

mode · validation · three-category-constraints
Purpose
Validate proposed mode configurations against policy, state-machine, and service-dependency constraints. Return ACCEPTED or REJECTED with violations.
Invocation
Synchronous from ModeService.save() and ModeService.activate(). No async path.
Input
{ modeId, modeDefinition, actorId, context } where context ∈ {SAVE, ACTIVATE}.
Output
{ status, violations[] }. Violations enumerated, not first-fail.
Hardcoded
Three-category constraint structure · synchronous from ModeService · rejection on any violation.
Operational engines (TimerEngine, PricingPipelineEngine, AvailabilityEngine, OverbookingDetectionEngine, CreditCeilingMonitorEngine, DisputeGateEngine, FOCValidationEngine, NightAuditEngine, RoomAssignmentSuggestionEngine) are consumed by admin services through their configuration surface dependencies. The Admin Console owns the configuration they read; it does not re-specify their behaviour.
